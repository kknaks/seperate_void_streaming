# Code Review Checklist

> 운영 체크리스트 — *어떤 순서로 무엇을 점검·실행·검증하는가* (SSOT).
> 룰의 본질 (왜 강제되는가) 은 `rules.md` 가 SSOT. 본 checklist 는 *실행 절차* 만 — 룰 본문 중복 박지 않음.
> 운영 점검 항목이 늘어나면 본 파일을, 룰 자체가 늘어나면 `rules.md` 를 갱신.

## Pre-flight

- [ ] 리뷰 대상 식별 — `/review` (git diff) / `/review path/...` (특정 파일) / `/review {domain}` 중 하나
- [ ] 사용처 프로젝트의 컨벤션 reference 로드:
  - `docs/common/*.md` 존재 여부 + 로드 가능 파일 목록
  - `CLAUDE.md` 존재 여부
  - 둘 다 없으면 → `rules.md` §reference 로드 모델 §fallback 작동 명시
- [ ] 변경 규모 측정 (총 +/- 줄 수)
  - 400 줄 초과 → 리포트 §맥락 에 ⚠ 분할 권장 표시

## Phase 1 — 맥락 파악 (1–2분)

- [ ] 변경 범위 (파일·도메인 목록)
- [ ] 도메인 의도 (기존 코드 패턴 vs 새 도입)
- [ ] 변경 목적 분류 (새 기능 / 버그 수정 / 리팩토링) — 커밋 메시지 / PR 설명 참조

## Phase 2 — 높은 수준 검토 (3–5분)

- [ ] 아키텍처 적절성 (계층 분리·책임 분배)
- [ ] 의존 방향 (외층 → 내층, 역방향 X)
- [ ] 컨벤션 reference 의 룰 위반 여부 (load 된 `docs/common/*` 의 큰 룰)
- [ ] 입출력 모델 분리 (요청/응답 vs 도메인)

## Phase 3 — 줄 단위 검토 (5–10분)

**언어 무관**:
- [ ] 보안: SQL 인젝션·민감 데이터 노출·권한 검증
- [ ] 성능: N+1·비효율 루프
- [ ] 명명: 도메인 어휘 일관성·의도 표현
- [ ] 타입: nullable·경계값

**언어 특화** (사용처 컨벤션 의존):
- [ ] (Python 예) 가변 기본 인자·bare except·타입 힌트·`X | None`
- [ ] reference 의 언어별 룰 항목

## Phase 4 — 요약 (1–2분)

- [ ] 심각도 5분류로 이슈 분배 (🔴 / 🟡 / 🟢 / 💡 / 🎉)
- [ ] 🎉 **praise 1건 이상** — 누락 시 차단
- [ ] 우선순위 결정 (🔴 → 🟡 순)

## 리포트 작성

- [ ] §맥락 (대상·규모·유형·reference 로드 결과)
- [ ] §요약 (심각도별 카운트)
- [ ] 각 이슈 — ID (`B-NNN`/`I-NNN`/`N-NNN`/`S-NNN`) + 파일:라인 + Convention 출처 + Before/After
- [ ] §🎉 잘된 점 (1건 이상)
- [ ] §다음 단계 (우선순위 가이드 또는 후속 작업)

## Post-flight

- [ ] 사용자에게 리포트 전달 (수정은 사용자 손)
- [ ] 시크릿 마스킹 확인 (`.env`·token 노출 X)
- [ ] (옵션) `/review` 재호출로 수정분 재검토
